Utility Computing

Definition
Utility computing = treating computing resources like a utility (electricity, water). You pay as you go and can dynamically provision virtual machines on demand.

Why it matters:

→ Scalability: theoretically "infinite" capacity — you can always rent more VMs.

→ Elasticity: scale up during peak load, scale down when idle. No wasted hardware.

Everything as a Service (XaaS)

Data Center Architecture

Core Idea
"The datacenter is the computer." Instead of thinking about individual machines, we treat the entire cluster of machines in a data center as one giant processing unit. Big data problems are distributed across all these machines.

Physical Building Blocks

A data center is organised hierarchically:

[Server] ─→ [Rack Switch] ─→ [Datacenter Switch]
~30-40 servers per rack connects racks in the DC

Each server contains DRAM (fast memory) and a hard disk (slow storage). Servers in the same physical rack are connected via a rack switch. Racks across the data center communicate via a higher-level datacenter (cluster) switch.

You need to know the hierarchy: Server → Rack → Datacenter. The further data travels across this hierarchy, the slower and more expensive it becomes.
03

Latency vs. Bandwidth

TermDefinitionAnalogy
Bandwidth Maximum data transmitted per unit time (e.g. GB/s). Describes capacity. The number of lanes on a highway
Latency Time for 1 packet to travel from source to destination (one-way), or round trip. Describes delay. The distance you must travel regardless of highway size
Throughput Actual rate at which data was transmitted over a period (as opposed to the theoretical max bandwidth). How much traffic actually flowed through the highway
Key insight
Adding more highway lanes (↑ bandwidth) does not reduce how long your journey takes (latency). These are independent! For small data, latency dominates. For large data, bandwidth dominates.

How Latency & Bandwidth Combine Along a Path

Rule 1 — Latency is additive
Total latency = sum of latencies across all hops.
Example: 1ms hop + 100ms hop = ~101ms total.
Rule 2 — Bandwidth is the minimum (bottleneck)
The path's bandwidth = the bandwidth of its slowest segment.
Example: 1 Gbps link → 10 Mbps link = ~10 Mbps effective bandwidth.

These are approximations — real networks are affected by congestion, transport protocols, and queueing — but they are the right model for this course.

Storage Hierarchy & Data Access Costs

Within a server, and across the datacenter, data can live in different storage media with very different characteristics:

Storage TypeLatencyBandwidthCapacityCost
DRAM (local) ~100 ns ~20 GB/s Low Expensive
Flash / SSD ~100 µs ~1 GB/s Medium Medium
Hard Disk (local) ~10 ms ~200 MB/s High Cheap
Key Observations
1. DRAM is fast but expensive and small. Disk is slow but cheap and large.
2. As you go further in the hierarchy (Local → Rack → Datacenter), capacity grows but latency increases and bandwidth drops.
3. Disk reads are ~100,000× slower in latency than DRAM reads (10ms vs. 100ns).

Sequential vs. Random Disk Access

This is critical for system design. The hard disk example makes this stark:

Access PatternPerformanceWhy?
Random access ~400 KB/s effective
(10ms latency per 4KB read)		 Each access requires the disk head to physically seek to a new location — that 10ms dominates.
Sequential access ~200 MB/s The disk head reads a continuous stream — no seeking needed, so full bandwidth is achieved.

→ Sequential is ~500× faster than random on disk. This is why "process data sequentially" is one of the four big ideas.

05

Data Flow Paths — The 4 Cases

These are the four standard data access scenarios you need to know. All calculations use the additive latency and bottleneck bandwidth rules.

Case 1 — Local DRAM (same machine)

My Program ─→ Local DRAM
Latency: ~100 ns
Bandwidth: ~20 GB/s

Fastest possible access. Everything lives on the same machine — no network involved.

Case 2 — Local Disk (same machine)

My Program ─→ Local Disk
Latency: ~10 ms
Bandwidth: ~200 MB/s

Still on the same machine, but now using the slow hard disk. Latency jumps by 100,000× vs. DRAM.

Case 3 — Remote DRAM in same rack

My Program ─→ Rack Switch ─→ Server B's DRAM
switch: 300µs, 100MB/s per port
Latency: ~600 µs (300µs × 2 hops)
Bandwidth: ~100 MB/s

The rack switch adds ~300µs each way (×2 for the two ports traversed). Bandwidth bottlenecks at the switch's 100 MB/s.

Case 4 — Remote DRAM in different rack

My Program ─→ Rack Switch ─→ DC Switch ─→ Rack Switch ─→ Server C's DRAM
DC switch: 500µs, 10MB/s per port
Latency: ~1600 µs (500×2 + 300×2)
Bandwidth: ~10 MB/s

Now we cross two rack switches and one datacenter switch. Latency adds up across all hops; bandwidth bottlenecks at the datacenter switch (10 MB/s — the slowest link).

For disk accesses (Cases 2, 3, 4 with disk): disk latency (~10ms) dominates over network latency. Disk in a different rack has roughly the same latency as local disk — the 10ms disk seek swamps the network delay.
ScenarioLatencyBandwidth
Local DRAM100 ns20 GB/s
Local Disk10 ms200 MB/s
Same-rack DRAM~600 µs~100 MB/s
Same-rack Disk~10 ms (disk dominates)~100 MB/s
Different-rack DRAM~1600 µs~10 MB/s

The Four Big Ideas

These are the core design principles that all major big data systems (Hadoop, Spark, etc.) are built on. Know them well.

01
Scale "Out", Not "Up"

Instead of buying one very powerful (and expensive) server, add many cheap commodity machines. Horizontal scaling (out) is more cost-effective and fault-tolerant than vertical scaling (up).

02
Seamless Scalability

System performance should scale linearly with the number of machines. If 1 machine takes 100 hours, 10 machines should take ~10 hours. This lets you "buy" more performance by adding nodes.

03
Move Processing to the Data

Network bandwidth is the bottleneck in a cluster. Rather than moving large datasets across the network to your program, move your program (code) to the machine where the data already lives. Code is tiny; data is huge.

04
Process Data Sequentially, Avoid Random Access

Disk seeks are very expensive (~10ms each). Designing systems to read data in large sequential chunks achieves ~200 MB/s. Random access degrades this to ~400 KB/s — a 500× penalty.

Seamless Scalability — Aggregated Bandwidth

One concrete reason scale-out works so well is that resources aggregate linearly:

# MachinesAggregated Disk BWAggregated DRAM BW
1200 MB/s20 GB/s
102 GB/s200 GB/s
10020 GB/s2 TB/s

Abstractions for Big Data Systems

Why We Need Abstractions: Google's Motivation

Consider indexing the entire web:

Back-of-envelope calculation
20+ billion web pages × 20 KB = 400+ TB of data.
1 machine reads at ~30 MB/s → would take ~4 months to scan the web once.
Solution: use thousands of machines in parallel — but that creates new problems.

Challenges of Parallel Processing

When you distribute work across many machines, you face:

ChallengeDescription
Machine FailuresWith 3,650 servers, you lose ~1 machine per day on average. Google reportedly had ~2.5M machines → ~700 fail daily! Systems must handle failures gracefully.
SynchronizationWorkers run in unpredictable order. Shared state requires mechanisms like semaphores, barriers, and condition variables — and bugs like deadlocks and race conditions are common.
Programming DifficultyConcurrent distributed code is extremely hard to write and debug. Without a framework, every team writes custom low-level solutions — a huge burden.

Barrier (Synchronisation Primitive)

Definition
A barrier is a synchronisation point: all processes/workers must stop and wait at the barrier until every process has reached it. Only then do they all proceed. Used to ensure workers finish one phase before starting the next.

The Solution: The Right Abstraction

The key insight is to separate what to compute from how it's executed:

Programmer writes: map() + reduce() functions

Execution framework (Hadoop / Spark) handles:
  → Task assignment to workers
  → Data replication & fault tolerance
  → Synchronization & barriers
  → Aggregating partial results

Think of this as the "instruction set" (API) of the datacenter computer. Just as you don't manually manage transistors when writing Python, you shouldn't manually manage threads and sockets when writing a big data job.

Divide and Conquer Pattern

"Work" (large dataset)
↓ Partition
[w1]   [w2]   [w3]
worker worker worker
↓ Combine
"Result"

The framework handles partitioning the work, assigning it to workers, collecting intermediate results, and merging them into the final answer.

Quiz Questions & Worked Answers

Quiz Q1

Server A is downloading a file from the disk of Server B in the same rack. What is the data path?

(A) Server B's DRAM → Rack Switch → Server A's DRAM
(B) Server B's disk → Server B's DRAM → Rack Switch → Server A's DRAM
(C) Server B's disk → Server B's DRAM → Rack Switch → Datacenter Switch → Rack Switch → Server A's DRAM

Answer: (B) — Data must first be loaded from B's disk into B's DRAM, then sent over the rack switch to A. Since A and B are in the same rack, there's no need for the datacenter switch (which is only needed for cross-rack communication).
Quiz Q2

Same scenario. If the file is very small, what determines how long this takes?

(A) Disk latency   (B) Disk bandwidth   (C) Network switch bandwidth   (D) Network switch latency

Answer: (A) Disk latency. For small files, the fixed seek time (~10ms) dominates. Bandwidth is irrelevant because there's very little actual data to transfer — the bottleneck is just the initial delay of getting the disk head to the right position.
Quiz Q3

Same scenario. If the file is very large, what determines how long this takes?

(A) Disk latency   (B) Disk bandwidth   (C) Network switch bandwidth   (D) Network switch latency

Answer: (C) Network switch bandwidth. For large files, latency is negligible compared to transfer time. The bottleneck is the rate of data transfer. Disk bandwidth is ~200 MB/s, but the rack switch is only ~100 MB/s — so the switch is the bottleneck (minimum of the two).
Quiz Q4

Program P on Server A in Rack 1. L1 = latency to local disk, L2 = latency to disk of another server in Rack 1, L3 = latency to disk of a server in Rack 16. Which are true?

S1. L3 can be 10× larger than L2.   S2. L2 can be 10× larger than L1.   S3. L3 ≈ L2.   S4. L2 ≈ L1.

Answer: S3 and S4. Disk seek time (~10ms) is so large that it completely dominates any network latency (which is in µs). So L1 ≈ L2 ≈ L3 ≈ 10ms. The hard disk is the bottleneck regardless of where in the datacenter that disk lives.
Takeaways
1. The datacenter is the computer — think at the cluster level.
2. Four big ideas: scale out, seamless scalability, move code to data, sequential access.
3. Hardware reality constrains design: disk is slow, network bandwidth is the bottleneck, DRAM is fast but limited.
4. The right abstraction (map/reduce) hides complexity from the programmer and enables reliable distributed computing.
